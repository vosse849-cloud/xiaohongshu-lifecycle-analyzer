# 项目总结：小红书笔记生命周期分析器

## 1. 项目定位

这是一个面向小红书内容运营的本地网页数据分析工具，也可以理解为 AIGC 内容运营效率工具。

它不是普通静态页面，也不是小游戏，而是围绕真实运营流程构建的 CSV 数据复盘工具。用户上传小红书后台导出的 CSV 后，工具会自动完成数据清洗、指标计算、状态判断、多快照增长对比和复盘辅助。

## 2. 解决的问题

小红书后台导出的数据通常是表格明细，运营人员仍然需要手动判断：

- 哪些笔记表现好？
- 哪些笔记只是普通？
- 哪些笔记可能冷启动失败？
- 哪些笔记可能进入低速增长？
- 多次导出的 CSV 之间，账号整体新增了多少观看、点赞、收藏？
- 哪些选题值得复用，哪些暂时不适合继续扩展？

这个项目把这些判断流程整理成一个本地工具，减少重复计算和人工筛选。

## 3. 核心解决方案

项目把 CSV 数据分成三个层次处理：

1. **基础数据层**：清洗 CSV 字段，解析发布时间、观看量、曝光、点赞、收藏、评论、分享、涨粉等指标。
2. **规则判断层**：根据观看量、赞藏率、发布时间、每小时新增等规则，判断作品状态和增长状态。
3. **运营复盘层**：用图表、表格和 AI 复盘文字帮助用户理解当前内容表现。

AI 只负责“复盘总结”，不替代基础数据计算，也不重新解析 CSV。

## 4. 已实现功能

- 单份 CSV 上传与分析。
- 小红书 CSV 表头兼容和字段清洗。
- 中文发布时间解析。
- 当前观察时间识别。
- 已发布时长计算。
- 赞藏和、互动量、赞藏率、收藏率、评论率、分享率、涨粉率等指标计算。
- 笔记状态判断：爆款、普通、失败、疑似尾流等。
- 表格排序、筛选、冻结列。
- 选题/菜品名自动识别。
- 选题手动编辑和 localStorage 修正库。
- 选题修正库导入导出。
- 多份 CSV 快照对比。
- 相邻快照新增数据计算。
- 增长判断：明显放量、稳定增长、小幅增长、疑似尾流、基本停滞等。
- 数据报表看板。
- 本地 Node 后端接入 DeepSeek API。
- 一键生成本地规则复盘和 DeepSeek AI 复盘。
- token 用量展示。
- Windows 一键启动、诊断和关闭脚本。

## 5. 功能迭代价值

项目从第一版静态 CSV 分析器逐步迭代：

1. 先解决真实 CSV 读取和字段兼容。
2. 再加入生命周期状态判断。
3. 然后加入选题识别和修正库，解决“原始标题不等于真实选题”的问题。
4. 再加入多快照对比，从单次状态判断升级到增长判断。
5. 加入图表看板，让运营复盘更直观。
6. 最后加入 DeepSeek AI 复盘，把结构化数据转成可复制的运营结论。

这个过程体现了基于真实反馈持续迭代的产品思路。

## 6. 运营价值

对内容运营来说，这个工具可以帮助：

- 快速判断当前账号整体表现。
- 定位爆款、稳定增长、疑似尾流和低反馈作品。
- 找到值得复用的选题和内容结构。
- 对多次导出的数据进行新增对比。
- 减少 Excel 手工计算和重复筛选。
- 把复盘结论沉淀成可复制文本。

## 7. AI 协同开发价值

这个项目适合作为作品集展示，因为它体现了：

- 能把模糊运营需求拆成明确功能。
- 能持续根据真实问题修复 CSV 兼容、时间解析、筛选联动、数据展示等细节。
- 能把 AI 放在合适的位置：辅助总结，而不是替代规则计算。
- 能注意本地隐私、API Key 安全和 GitHub 上传前的安全检查。

开发过程中使用 Codex 辅助完成代码实现和迭代，但产品方向、业务场景、判断规则、验收标准和每一轮优化需求由我定义。这个项目更想展示的是：我能把真实内容运营问题拆成可落地的工具需求，并持续验证、修正和收敛到可展示版本。

## 8. 当前限制

- 当前仍然依赖用户导出 CSV。
- 真实尾流判断需要连续多次快照验证，单次数据只能判断“疑似”。
- 选题自动识别依赖词库和规则，不可能百分百准确。
- AI 功能需要用户本地配置 DeepSeek API Key。
- 目前没有完整在线 mock 演示模式。
- 真实小红书 CSV 不建议上传 GitHub。

## 9. 后续优化方向

- 增加连续多区间趋势判断。
- 支持更多图表。
- 增加脱敏数据模板和演示数据生成器。
- 做成桌面 App，进一步降低小白使用门槛。
